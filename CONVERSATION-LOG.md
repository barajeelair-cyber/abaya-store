## 1. البداية ـ المشروع الأساسي

**الطلب الأصلي**: متجر إلكتروني لـ "العبايات" يستهدف مدن قطاع غزة.

**التقسيم**:
- ✨ صفحة الزبائن (Customer Front-End) - أسود فاخر + ذهبي
- ⚙️ لوحة تحكم الإدارة (Admin Dashboard) - أبيض لؤلؤي + أسود

**التقنيات المختارة**: HTML + CSS + Vanilla JS (بدون build step)، يفتح مباشرة من المتصفح أو يُنشر على GitHub Pages.

**الملفات الأولية**:
```
abaya-store/
├── index.html              ← المتجر
├── admin.html              ← الإدارة
└── assets/
    ├── css/customer.css + admin.css
    └── js/data.js + customer.js + admin.js
```

**طبقة البيانات** (`data.js`):
- `ProductsAPI` / `OrdersAPI` / `SettingsAPI`
- التخزين: `localStorage` بمفتاح `abaya_amal_v1`

**المدن المضمنة**: 12 مدينة فلسطينية (غزة، خانيونس، رفح، النصيرات، البريج، المغازي، إلخ).

## 5. إصلاحات حرجة

### 🔴 إصلاح TDZ Error في admin.js (refresh يكسر الموقع)
**السبب**: `if (AuthAPI.isLoggedIn()) showApp()` في السطر 16 كان يستدعي `fillCategorySelects` التي تستخدم `$filterCat` المُعرَّف في السطر 280 (Temporal Dead Zone).

**الإصلاح**: نقل فحص الجلسة إلى نهاية الملف (بعد كل الـ `const` declarations).

### 🔴 إصلاح Service Worker العالق
- تحويل SW إلى passthrough (لا يخزّن أي شيء).
- إضافة آلية auto-reload بعد التفعيل.

### 🔴 إصلاح localStorage الموروث
- الترحيل القديم كان "إضافة فقط" → الزبائن القدامى يبقون مع البيانات القديمة.
- إصلاح: كشف صيغة قديمة (مثلاً sizeCharts بدون عمود sleeve) → استبدال بالافتراضي الجديد.

## 8. إصلاحات Supabase الجذرية

### 🔥 المشكلة: أزرار الإدارة لا تحفظ
**السبب الجذري**: `data-supabase.js` الأول كان يعيد تعريف كل APIs كـ **async/Promise**، لكن `admin.js` كان يستدعيها **sync**:
```js
const products = ProductsAPI.list();   // Promise
products.filter(...);                  // TypeError → silent fail
```

### ✅ الإصلاح: Sync-Cache Architecture
1. عند بدء التطبيق، تحميل كل البيانات الأولية في cache واحد عبر `Promise.all`.
2. بعد جاهزية الـ cache، إطلاق حدث `data-ready`.
3. **القراءة sync**: `list()` و `get()` ترجعان من cache مباشرة.
4. **الكتابة optimistic-sync**: `save()` و `remove()` تحدّث الـ cache فوراً + ترسل لـ Supabase في الخلفية.
5. customer.js و admin.js ينتظران `data-ready` قبل أول render.

### 🐛 إصلاحات إضافية
- `Utils.categoryById` و `Utils.cityById` يقرآن من `CategoriesAPI`/`CitiesAPI` (يدعم Supabase override).
- صورة المنتج: `p.colors?.[0]?.image` → `ProductsAPI.coverImage(p)` (يدعم الهيكل الجديد `images[]`).
- حد رفع الصور: 3 ميجا → **10 ميجا**.
- معالجة auth handlers لتعمل sync و async.

## 10. القرارات المعمارية

### ✨ لماذا Vanilla JS؟
- بسيط، يفتح مباشرة من المتصفح.
- لا يحتاج build step.
- يعمل على GitHub Pages بدون CI/CD.

### ✨ لماذا Sync-Cache فوق Async API؟
- `admin.js` كُتب أصلاً بـ sync style.
- إعادة كتابة كل handlers ليكونوا async = مخاطرة عالية.
- الحل: keeps API surface sync مع writes تفاؤلية.

### ✨ لماذا 2 manifest؟
- `manifest.json` للزبون (theme dark + RTL).
- `admin-manifest.json` للإدارة (theme light).
- iOS لا يحترم theme_color جيداً، لكن Android يحترم.

### ✨ لماذا قاعدة Supabase + GitHub Pages؟
- Supabase = قاعدة بيانات + Storage مجانية حتى 8GB.
- Vercel = نشر مع CDN + HTTPS تلقائي.
- معاً = backend بدون خادم خاص.
